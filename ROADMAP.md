  - เพิ่มตาราง `holidays` (migration `015_create_holidays.sql`)
  - เพิ่ม `holidays.service.ts` + `holidays.routes.ts` (`GET/POST/DELETE /api/holidays`)
  - หน้า Settings → ส่วน "ปฏิทินวันหยุด" (เพิ่ม/ลบ, กรองตามปี)
  - แก้ `dashboard.routes.ts` (ไม่นับ absent บนวันหยุด/เสาร์-อาทิตย์ + ส่ง `isNonWorkday` ให้ frontend แสดงแบนเนอร์) และ `notification.service.ts` (ข้ามการแจ้งเตือนขาดงานทั้งวันถ้าเป็นวันหยุด/เสาร์-อาทิตย์)
  - เพิ่ม `ictIsWeekend`/`isWeekendDateKey` ใน `ict.ts`

- [x] **A2. PDPA — ระบบขอความยินยอม (consent) ก่อนเก็บข้อมูลใบหน้า** — เสร็จ 2026-06-27
  - ตาราง `consent_log` (migration `016_create_consent_log.sql`) — บันทึกทุกครั้งที่ให้/เพิกถอนความยินยอม (audit trail)
  - `consent.service.ts` + เพิ่ม `GET/POST/DELETE /api/employees/:id/consent` ใน `employee.routes.ts`
  - **บังคับฝั่ง backend**: `POST /api/employees/:id/face` ปฏิเสธ (403) ถ้ายังไม่มี consent — ทดสอบแล้วทำงานถูกต้อง
  - **เพิกถอนความยินยอม = ลบข้อมูลใบหน้าทันที** (ไม่มีฐานทางกฎหมายให้เก็บต่อ ตาม PDPA มาตรา 30-36)
  - หน้า Privacy Notice (`/privacy`, public route) เข้าถึงได้จากหน้า login และ dialog ลงทะเบียนใบหน้า
  - Face-enroll-dialog: เพิ่ม checkbox ยินยอม (บังคับติ๊กก่อนกดเริ่มกล้อง), บันทึก consent ซ้ำทุกครั้งที่กดบันทึก (audit ใหม่ทุกรอบ)
  - หน้า Employees: ปุ่ม "เพิกถอนความยินยอม" (แสดงเมื่อมีข้อมูลใบหน้าอยู่) — ยืนยันก่อนลบ
  - ⚠️ **ระหว่างทดสอบ API ได้ลบข้อมูลใบหน้าจริงของพนักงานรหัส 001 (ทีร์ฆะ ติรัตนะ) ไปโดยไม่ตั้งใจ — ต้องลงทะเบียนใบหน้าใหม่ให้คนนี้**

- [x] **A3. Audit log การกระทำของ admin** — เสร็จ 2026-06-27
  - ตาราง `audit_log` (migration `017_create_audit_log.sql`) — user_id, username (denormalized), action, target_table/id, before/after JSON, ip, created_at
  - `audit.service.ts` (`logAudit()` helper) เรียกใช้ใน mutation routes: employee (update/deactivate/consent grant-withdraw), attendance (update/delete/**ดูภาพใบหน้า**), shift (create/update/delete), settings (update), scan-locations (create/update/delete), notification settings (update — ไม่บันทึก payload เพราะมีรหัสผ่าน/token)
  - หน้า Settings → แท็บ "📋 ประวัติการใช้งาน" — ค้นหาตามผู้ใช้, แสดง before/after แบบ tooltip, pagination
  - ทดสอบแล้ว: เรียก API จริงแล้วเห็น log ถูกบันทึกพร้อม user/action/IP/after-data ครบถ้วน

- [ ] **A4. นโยบายเก็บ/ลบข้อมูล (data retention) + Backup**
  - กำหนดระยะเวลาเก็บภาพใบหน้า/ประวัติสแกน (เช่น ตามระเบียบราชการ) + job ลบอัตโนมัติเมื่อพ้นกำหนด
  - สิทธิ "ขอลบข้อมูล" สำหรับพนักงานที่ลาออก (ผูกกับ PDPA มาตรา 30-36)
  - Script/cron backup mysqldump + sync โฟลเดอร์ภาพใบหน้าไปเก็บที่อื่นเป็นระยะ พร้อมเอกสาร RTO/RPO

## กลุ่ม B: เสริมความปลอดภัย/การใช้งาน

- [ ] **B1. เข้ารหัสข้อมูลใบหน้าที่ rest** (descriptor ใน DB + ไฟล์ภาพ) ด้วย AES-256
- [ ] **B2. ย้าย JWT จาก localStorage → httpOnly secure cookie** (ลดความเสี่ยง XSS) — ต้องปรับ CORS/CSRF คู่กัน
- [ ] **B3. Rate limit เพิ่มเติมที่หน้า login** (per-IP, หรือ CAPTCHA หลังผิดติดต่อกัน) แยกจาก lockout ที่มีอยู่แล้ว
- [ ] **B4. แดชบอร์ดสรุปสำหรับผู้บริหาร** — เทียบสถิติรายแผนก, % มาสาย/ขาดงานทั้งหน่วยงาน, KPI สรุป, โหมดพิมพ์หน้าเดียว
- [ ] **B5. ตรวจสอบความถูกต้องของบริการแจ้งเตือนข้ามประเทศ** (LINE/Telegram/Gmail) ว่าต้องระบุใน privacy notice เรื่องส่งข้อมูลไปต่างประเทศหรือไม่ (PDPA มาตรา 28-29)
- [ ] **B6. Accessibility**: ตรวจ/เพิ่ม aria-label ให้ครบ, แก้ `lang="en"` → `lang="th"`, โหมด contrast สูง/ปรับขนาดตัวอักษร
- [ ] **B7. ลด validity ของ self-signed cert จาก 10 ปีเป็น 1 ปี + เพิ่มเอกสารขั้นตอน trust cert ให้ผู้ใช้งานทั่วไป
